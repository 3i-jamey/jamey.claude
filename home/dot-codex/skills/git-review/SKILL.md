# 리뷰: $ARGUMENTS

## 워크플로우

1. **리뷰 대상 결정**:
   - `$ARGUMENTS`에 파일/경로가 있으면 그것만 리뷰.
   - 없으면 미커밋 전체: `git diff HEAD` + `git status`의 미추적 파일.
   - 워킹 트리가 깨끗하면 최근 커밋 폴백: `git show HEAD`.
2. **변경 규모 점검**: 파일 10개 이상 또는 단일 diff 300줄 이상이면 사용자에게 "대규모 변경이라 리뷰가 길어진다"고 알리고 우선순위(영향 큰 파일부터)를 잡는다.
3. **변경된 파일 전체 읽기**.
4. **호출 관계 / 히스토리 파악**:
   - 변경된 함수·클래스가 어디서 호출되는지 grep (`grep -r "<symbol>" --include='*.<ext>'`).
   - 변경된 라인 부근 `git blame <file> -L <start>,<end>`로 의도·최근 수정 빈도 확인.
5. **테스트 매칭 확인**: 변경된 파일 옆/대응되는 테스트 파일이 있는지 보고, 동작 변경에 테스트가 따라왔는지 본다.
6. **프로젝트 규칙 로드**: `AGENTS.md` / `CONTRIBUTING.md`에 명시된 아키텍처·코딩 규칙을 체크리스트에 합쳐 적용한다.
7. 아래 체크리스트(이 파일 하단)를 수행한다.
8. 마무리 규칙을 참고하여 리턴 형식대로 출력 후 스킬 실행을 마친다. 사용자 응답은 받지 않는다 (보고 전용 스킬).

## 반드시 지킬 규칙

- 발견만 하고 자동으로 고치지 않는다.
- 코드 수정·커밋·푸시 같은 변경 행위를 하지 않는다. 허용되는 git 명령은 읽기 계열(`git status`, `git diff`, `git log`, `git show`, `git blame`, `git fetch`)뿐.
- 변경이 없으면 그대로 보고하고 이슈를 지어내지 않는다.
- `AGENTS.md` / `CONTRIBUTING.md` 명시 규칙 위반은 항상 **🚨 Critical**로 표기한다.
- diff만 보지 않고 변경된 파일 전체를 읽는다.
- 추측이 아닌 실제 코드를 인용해 지적한다. `<file>:<line>`과 코드 스니펫을 함께 제시한다.

## 기본 규칙

- 잘된 점도 최소 1개는 언급한다.
- 심각도는 **🚨 Critical / ⚠️ Major / 💡 Minor** 셋만 사용한다.
- submodule pointer 변경이 들어 있으면 별도로 표시한다.
- 가능하면 외부 베스트 프랙티스 링크를 1~2개 첨부한다. 추측 링크는 금지.

## 마무리 규칙

- 검증: 변경이 전혀 없고 최근 커밋도 없는 경우 → `BLOCKED: NO_CHANGES`와 한 줄 사유로 종료.
- 보고: 변경된 파일 본문·blame raw·grep 결과·중간 추론은 헤딩 위쪽에 두거나 출력 안 함. 최종 응답에 포함하지 않는다.
- 결과: 아래 출력 포맷을 리턴 형식대로 둔다. 사용자 확인 안내는 붙이지 않는다.

## 체크리스트

**잔여물:**
- [ ] 이번 변경에 추가된 디버그 로그(`Log.*`, `console.log`, `print`, `dbg!`, `debugger`, `Timber` 임시), `TODO/FIXME` 없음
- [ ] 주석 처리된 코드 블록 없음
- [ ] 테스트에 `@Ignore`/`.only`/`.skip`/하드코딩된 인증정보 없음

**비밀값/설정:**
- [ ] diff에 API 키, 토큰, 비밀번호, 내부 URL 없음
- [ ] `.env`, `local.properties`, 키스토어/`*.pem`/`*.key` 같은 인증 파일이 스테이징되지 않음
- [ ] 새 환경변수·플레이버 설정은 팀이 볼 수 있는 곳에 문서화됨

**프로젝트 규칙 (AGENTS.md / CONTRIBUTING.md):**
- [ ] 명시된 아키텍처 위치 규칙(레이어/모듈 위치) 준수
- [ ] 명시된 비동기/동시성 규칙 준수
- [ ] 명시된 null 안전성/타입 안전성 규칙 준수
- [ ] 명시된 라이프사이클/자원 관리 규칙 준수
- [ ] 명시된 의존 방향(예: `presentation → domain → data`) 위반 없음
- [ ] 명시된 베이스 클래스/패턴 사용

**안정성/사이드이펙트:**
- [ ] null이 될 수 있는 LiveData/StateFlow/Optional/nullable 경로 처리됨
- [ ] 컬렉션·배열 접근 시 범위 검증 누락 없음
- [ ] 멀티스레드/코루틴 접근에서 동시성 위험 없음 (공유 가변 상태, 잘못된 dispatcher)
- [ ] 라이프사이클 경계(detach/destroy 후 context 접근, 잘못된 scope) 위반 없음
- [ ] 공통/공유 모듈 변경이 호출자에게 미치는 영향 검토됨
- [ ] 공개 API(인터페이스/시그니처/data class 필드) 깨짐 없음

**정확성:**
- [ ] 리네임/삭제 전파됨. 삭제된 심볼·문자열 리소스 참조 없음
- [ ] import 깨끗함 (미사용 없음, 누락 없음, wildcard 남용 없음)
- [ ] 에러 처리는 시스템 경계(API, DB, BLE, 카메라 등)에서만. 내부 에러 삼키지 않음
- [ ] UI 프레임워크 패턴(Compose `remember`/state hoisting, MVI 상태 전이 등) 일관 사용

**성능:**
- [ ] 메인 스레드에서 IO/네트워크/무거운 연산 없음
- [ ] 대용량 리소스(Bitmap, 큰 리스트) 미해제·전체 로딩 없음
- [ ] 반복문 안 쿼리/객체 생성 같은 명백한 핫스팟 없음

**범위:**
- [ ] 모든 hunk가 해당 티켓·작업 의도에 기여. 곁다리 리팩토링 없음
- [ ] 호출 지점이 하나뿐인 추상화를 새로 만들지 않음
- [ ] 투기적 기능/플래그/설정 노브 없음

**테스트:**
- [ ] 동작 변경에 테스트 변경 따라옴 (UseCase, ViewModel 단위 테스트 우선)
- [ ] 코드 실행만 확인하는 테스트가 아니라 실제 동작·상태 전이를 검증
- [ ] 가능하면 로컬 테스트 스위트 실행 결과 확인

**문자열/리소스:**
- [ ] 새 사용자 노출 문자열은 리소스 파일(`strings.xml` 등)에 있음 (하드코딩 금지)
- [ ] 다국어 키가 한쪽 언어에만 들어가지 않음

## 출력 포맷

```
## 로컬 리뷰

### 판정: [PASS / PASS WITH FLAGS / FAIL]
- 변경 파일: <개수>
- 변경 규모: <대략 줄 수 / 또는 "소규모">

### 이슈
- 🚨 **Critical** <파일>:<줄> — <무엇이 잘못됐고 왜 중요한지> · 제안: <…>
- ⚠️ **Major** <파일>:<줄> — <…> · 제안: <…>
- 💡 **Minor** <파일>:<줄> — <…> · 제안: <…>

### 잘된 점
- <파일>:<줄> — <어떤 결정·패턴이 좋았는지>

### 통과
- 깔끔하게 통과한 체크 요약

### 제안 (블로커 아님)
- <…>

### 참고 (선택)
- <외부 베스트 프랙티스 링크 1~2개>
```
